四、Vibe Slop 风险判断
当前项目存在一定 vibe slop 风险，但不是不可控状态。
原因是功能增长较快，多次以增量方式叠加 UI、状态、OCR、AI调用、排行榜、卡片等逻辑，静态检查能证明当前主要连接关系没有断，但还不能等同于工程质量已经稳定成熟。

五、识别出的主要风险与改进建议
- [中] 前端单文件过大：renderer.js 当前约 64801 字符，功能持续堆叠在一个文件中。 建议：下一阶段应拆分为 projects、leaderboard、analysis-cards、settings、utils 等模块。
- [中] JSON本地存储容量与一致性风险：候选人数量很大时，单个 projects.json 会变大；异常退出时存在写入中断风险。 建议：候选人超过数千条后升级 SQLite；写入采用临时文件+原子替换。
- [中] OCR打包与识别质量风险：tesseract.js 在 GitHub Actions 打包和 Windows 安装后可能受到 worker/语言包路径、截图清晰度影响。 建议：安装后必须实测 PNG/JPG；必要时固定 workerPath/langPath，或改成独立OCR服务/可选OCR通道。
- [低] 状态保存点较多：多处 UI 操作都会保存项目，逻辑可用但维护成本上升。 建议：后续统一 saveProjectState 函数，避免某个新按钮漏保存。
- [中] 候选人卡片与排行榜复用同一数据结构：候选人卡片、排行榜、项目存储共用 leaderboard 数据，短期节省结构，但长期语义会混乱。 建议：下一版可把 candidateCards 独立为 candidates 表/数组，leaderboard 只作为视图或排序结果。
- [中] API Key 本地明文保存：接口密钥会写入本机 settings.json，便于单机使用，但不是系统级加密保存。 建议：后续使用 Electron safeStorage 或系统钥匙串保存；完整备份继续默认不导出密钥，并保留二次确认。

六、下一阶段建议
1. 不再继续大量堆新功能，先做工程整理。
2. 拆分 renderer.js 和 main.js。
3. 加入基础自动化测试。
4. 使用 SQLite 替代大型 JSON 文件。
5. 使用 safeStorage 或系统钥匙串保存 API Key。
6. 实机测试 OCR 打包后的可用性。
7. 将候选人卡片与排行榜数据结构分离。
